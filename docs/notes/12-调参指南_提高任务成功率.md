# 12 · 调参指南 —— 提高任务成功率

按优先级排，前面收益大。

## 第一原则：先看 metric，再动参数

### 关键指标速查

| 指标 | 看什么 |
|---|---|
| `rollout/rewards` & `returns_mean` | **最关键**，长期单调上升；100 epoch 不动 = 没在学 |
| `train/actor/value_loss` | 前期震荡，后期应降；一直爆炸 = advantage/GAE 有问题 |
| `train/actor/policy_loss` | 不能 NaN/Inf；正负震荡是 PPO clipped surrogate 设计，**正常** |
| `train/actor/entropy` | 应缓降；下降太快 = collapse |
| `approx_kl` / `kl_mean` | < 0.02；> 0.05 步太大 |
| `clip_fraction` | > 0.3 步太大 |
| `grad_norm` | 见阈值表 |
| `eval/success_rate` | 最权威，更新慢 |

只看 reward 不够。reward 跌 + entropy 跌 = collapse；reward 涨 + entropy 跌过快 = 早熟。

### 哪些震荡是设计行为，别误判

| 指标 | 为什么震荡 | 真异常 |
|---|---|---|
| `advantages_mean` | 开 `normalize_advantages` 后归一化到 mean=0, std=1 | 远离 0 或 std→0 |
| `policy_loss` | clipped surrogate 本就在 0 附近 | NaN/Inf 或单调走 |
| `value_loss` | returns 涨 → value 目标在动 | 持续单调升不收敛 |
| `kl` (`kl_beta=0`) | 只是观测量不进 loss | 均值 > 0.05 且涨 |
| `entropy_loss` | bonus 很小，噪声主导 | 突崩到极低 = collapse |

**套路**：震荡 + returns 涨 = 健康；震荡 + returns 跌 = 真有问题。

### `grad_norm` 阈值

`clip_grad: 1.0` 时 tb 记录 **clip 前**值：

| grad_norm | 判断 | 动作 |
|---|---|---|
| < 1.0 持续升 | 正常，policy 变 confident | 继续 |
| 1.0-5.0 震荡升 | 频繁被 clip，能跑 | 可接受 |
| > 10 持续升 | **危险**，可能塌缩 | 降 lr / 紧 clip |
| 突然 spike 回不来 | **马上停**，回退 ckpt | 检查 value_loss |

无害升高常见原因：`critic_warmup_steps` 结束、policy 变 sharp、LoRA + bf16 噪声尖刺。

### 小 batch 是噪声放大器

`total_num_envs=2`、`batch_size=8`、`update_epoch=4` 这种配置下，曲线比 PPO 论文图抖得多是统计噪声不是 bug。提到 `total_num_envs=8/16` 曲线就平滑了。

### 健康度黄金五条（每隔几十 step 扫）

1. ✅ `returns_mean` 长期向上
2. ✅ `approx_kl` < 0.02
3. ✅ `grad_norm` < 5（严格 < 1）
4. ✅ `clip_fraction` < 0.3
5. ✅ `entropy` 缓降（不是悬崖跌）

任何一条违反且持续多个 step → 看下面"症状-改什么"。

## 排序：先这样调

### 1. 检查 reward shape（最容易忽略）

打开 `tasks/so101_pick_place.py` 看 `evaluate()`：

```
reward = reach*0.3 + grasp*2 + move*1.5 + on_plate*2 + success*5
```

调试：跑 10 个 random episode 记每段平均拿多少。reach 占 90% → 模型可能"骗"reach 不抓 → reach 降到 0.1。grasp 完直接 success 没经过 move → move 信号太弱 → 加大 move 或加 distance shaping。

reward 是 RL 的"地图"，画错怎么走都到不了。

### 2. 起步学习率别太大

```yaml
optim:
  lr: 5.0e-6        # actor LoRA
  value_lr: 1.0e-4  # value head
```

LoRA RL 不是 SFT，5e-6 已经偏激进。前 100 step KL > 0.05 降到 2e-6。value head 新 init，1e-4 合理。

**信号**：前期 reward 突然崩到比初始 SFT 还低 → lr 太大，rollback。

### 3. `clip_ratio` 别乱开大

PPO 默认 0.2。flow VLA chunk-level ratio sum 出来方差更大，0.2 偶尔过紧，可试 0.3，但不超 0.4。

### 4. `update_epoch` 别贪心

PPO 论文 4-10。3090 显存吃紧 4 已经够。开 10 后几个 epoch ratio 漂得厉害（off-policy），收益递减伤稳定。

### 5. GAE 参数

```yaml
gamma: 0.99
gae_lambda: 0.95
```

PPO 默认值，一般不动。任务 > 500 step 提到 0.995。

### 6. `entropy_bonus` 平衡杆

```yaml
entropy_bonus: 0.005
```

观察 `train/entropy`：
- 一直 > -10 不下来 → 探索过多 → 降到 0.001
- 50 epoch 跌到 < -50 → collapse 风险 → 提到 0.01

### 7. `value_clip` / `huber_delta`

```yaml
value_clip: 0.2
huber_delta: 10.0
```

value 不稳时 Huber 比 MSE 抗噪。`value_loss` 周期性尖刺 → 降 `huber_delta` 到 5 或 2。

### 8. `max_episode_steps`：覆盖真机 q99 + 整除 chunk_size

两条约束：
1. **必须能被 `chunk_size`（50）整除**，见 [11 #12.5](11-踩坑总结.md)
2. **必须覆盖真机 q99**（RL 比 demo 慢，q99 + margin）

```python
sd = load_file('<lora_ckpt>/policy_preprocessor_step_5_normalizer_processor.safetensors')
q99 = sd['frame_index.q99'].item()
# max = ceil(q99 × 1.2 / chunk_size) × chunk_size
```

例：q99=308 步 → 1.2× = 370 → 上取整 50 倍 = **400 步**。

加大代价：每 epoch rollout 线性增加（200→400 总时长 +50%）。显存基本不变。

## 症状 → 改什么

| 症状 | 大概率原因 | 优先尝试 |
|---|---|---|
| reward 100 epoch 不涨 | exploration 不够 / reward 太稀疏 | ↑ entropy_bonus，查 reward shape |
| reward 饱和某 X | 卡某 stage | 看 per-stage 拆分，调权重 |
| reward 突然崩 | lr 太大 / clip 太松 | ↓ lr，restore ckpt |
| entropy 跌太快 | collapse | ↑ entropy_bonus，降 update_epoch |
| KL > 0.1 | step 太大 | ↓ lr，紧 clip，降 update_epoch |
| value_loss 一直高 | value head 跟不上 | ↑ value_lr，加 critic_warmup |
| 动作扭曲/撞机座 | RL 漂离 SFT | 混 SFT loss，或 KL penalty `kl_beta: 1e-3` |
| eval 比 train 差 | exploration noise 太大 | `temperature_eval: 0.0` |

## envs / batch 关系

```yaml
env.train.total_num_envs: 8
actor.global_batch_size: 8
actor.micro_batch_size: 1
```

- `total_num_envs` ≤ `global_batch_size`
- `global / micro = gradient accumulation steps`
- 3090 micro 必须 1。提多样性：加 envs 或 `rollout_epoch: 2` 多步 rollout

## 真不动了

按顺序换：

1. **加 ckpt resume** 保已有进展
2. **降难度**：固定方块位置 / 缩小 workspace / 加大方块尺寸；通了再加随机
3. **加初始权重**：SFT 模型先 BC 几步拉到 reward > 0 再开 RL（warm start）
4. **换 reward**：dense → sparse + success bonus，避免被 shaping 骗
5. **加 demonstration**：真机数据 pre-fill replay buffer，offline PPO 变体

## PPO vs GRPO

我们用 PPO 因 reward 多阶段稀疏（需要 GAE 做 per-step credit）。切纯 sparse reward 用 GRPO 更省事（不需要 value head，group-norm baseline）：

```yaml
algorithm:
  adv_type: grpo
  loss_type: ratio
  group_size: 8
actor.model.with_value_head: False
```
