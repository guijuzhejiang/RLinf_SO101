**Q：项目简介？**

两段：
1. **LeRobot 上 SmolVLA 的 LoRA SFT**：仿真 + 真机遥操混合数据，把 motion prior 学到位
2. **RLinf 接 PPO 后训练**：纯仿真（ManiSkill3 + SO-ARM 101），任务把红方块抓到白盘子；先跑通仿真再考虑真机

技术栈：PyTorch + FSDP（单卡 no_shard）+ PEFT LoRA + flow-matching VLA。

**Q：RLinf 里有哪几个角色？**

4 个：environment、rollout、actor、critic。先统一术语：

- **trajectory**：env 从 reset 到 done 的 step 序列，本配置 `max_episode_steps=200`
- **return**：从某步开始往后所有 reward 的**带折扣累加**

接下来 4 角色：

- **Environment**：仿真器（ManiSkill3）。吃 action，跑物理 + 渲染，吐 obs + reward
- **Rollout**：跑 VLA 推理。obs → action chunk → env，循环。trajectory 记 5 量：
  - `obs / action`：env 和模型来回
  - `reward`：env 算好返回
  - `logprob`：**采样时顺手记**。flow-matching VLA 这里是 SDE 注入那步给 Gaussian noise `eps` 算的 log p，按 chunk sum 成标量。**PPO 后面算 `ratio = π_new / π_old`，没 logprob 就没 ratio**
  - `value`：rollout 同时让 critic forward 一次，对每个 obs 算 V(s) 存下来。算 GAE advantage 用
- **Actor**：训练端，按 PPO 公式更新 LoRA。`total_num_envs=8`、`max_episode_steps=200`、`chunk_size=50` → 一轮 rollout 产 `8×200=1600` step-level transition = `8×4=32` chunk-level 决策；`global_batch_size=8` 一次更新吃 8 chunk
- **Critic**：value head 挂 VLA backbone 后

**critic 网络结构**（追问细节）：

不是独立网络，是 VLA backbone（SmolVLA 是 action expert transformer）后接 MLP：`expert_hidden → 512 → 128 → 1`，3 个 `Linear`，激活 **ReLU**，hidden Kaiming init，末层 `normal(std=0.02)`。输入是 `suffix_out[:, :chunk_size]` 沿 chunk 维 **mean pooling** 得 `[B, hidden]`，输出 `[B]` V(s)。代码 [value_head.py](rlinf/models/embodiment/modules/value_head.py)。

**critic 怎么训**：
- **target**：不是 reward 本身，是 **GAE return = advantage + V_old(s)**
- **loss**：带 clip 的 **Huber**，`value_clip=0.2, huber_delta=10.0`：
  ```
  V_clipped = V_old + clip(V_new − V_old, ±0.2)
  critic_loss = max( Huber(R − V_new), Huber(R − V_clipped) )
  ```
- **评估**：`explained_variance = 1 − Var(R−V)/Var(R)`，越接近 1 越好。我接 SmolVLA 第一次跑 EV 长期是 nan，定位到不是 critic 坏是 input pipeline 没接通（详见后面 silent bug 题）

**reward ≠ V(s)**：reward 是 env 给的**当前一步**即时奖励；V(s) 是 critic 估计的**从这步往后所有未来 reward 期望累加**。本任务 reward 5 阶段稠密见 [05](05-自定义PickPlace任务.md)。

**return / advantage 举例**：trajectory 前 4 步 reward `[0.1, 0.1, 0.1, 2.0]`，`γ=0.99`：
- 第 1 步 return = `0.1 + 0.99·0.1 + 0.99²·0.1 + 0.99³·2.0 ≈ 2.24`
- critic 估 V(s₁)=1.8 → **advantage = 2.24 − 1.8 = 0.44 > 0** → "实际比预期好，鼓励这次 action"
- 若 V(s₁)=2.5（高估）→ advantage = −0.26 < 0 → "抑制"

**一句话总结**：env 给 reward，rollout 攒 traj+logprob+value，critic 估 V 做 baseline，actor 按 advantage 调策略，新权重 sync 回 rollout。

**常见误解**：PPO 不是"对成功轨迹强化"，是**对高 advantage 的 action 强化**。同 reward=1，V(s)=0.9（容易）和 V(s)=0.1（难），后者 advantage 大得多。**没 critic 就没法区分"运气好"和"策略好"**。

---

**Q：为什么 PPO 不 GRPO？**

reward 是多阶段稠密（reach/grasp/move/on_plate/success），PPO 的 value head + GAE 能做 per-step credit assignment。GRPO 的 group-norm baseline 在长视距稠密 reward 上不如 GAE 准。切纯 sparse reward 时 GRPO 反而省事。

---

**Q：LoRA RL 跟 SFT 不一样在哪？**

- 同 FSDP unit 里 base `requires_grad=False`，LoRA `requires_grad=True`，要开 `use_orig_params=True`
- LoRA RL lr 比 SFT 低一个量级（SFT 1e-4，RL 1e-5~1e-6）。LR 大了漂离 SFT motion prior
- 显存 LoRA 远小于 full FT，3090 24G 能跑 SmolVLA 450M base + 1.5M LoRA
- weight sync 只 sync LoRA delta（base 不变），省通信

**Q：tensor shape 不匹配怎么排查？**

1. 看完整 stack trace，定位 `op(a, b)` 报错行
2. `print(a.shape, b.shape)` 或 debugger，弄清实际 shape
3. 反推应该是什么 —— 从 batch_size / seq_len / action_dim 推
4. `[B, T, A]` vs `[B, A]` broadcast 问题 → 多半是某步该 `unsqueeze` 没 unsqueeze。我踩过 `prev_values` 该 `[B, 1]` 出成 `[B]` 导致下游 `[:, :1]` IndexError
5. `[B*T, A]` vs `[B, T, A]` → 多半 reshape 没 reshape 回去

**绝对不要靠"猜一个 reshape 试试"**。先想清楚每一维代表什么。

**Q：flow-matching VLA 的 log_prob 怎么算？跟 token-AR 区别？**

token-AR（OpenVLA）：每生成一 token 一次决策，log_prob 是每 token softmax 取 log，整 seq 加起来。

flow-matching（SmolVLA / OpenPI）：不是逐 token 生成，没"概率"概念。做法是把 SDE 注入那步加 Gaussian noise，记采样时的 mean 和 std，用 Gaussian log_prob `-0.5*((eps-mean)/std)² - log(std) - 0.5*log(2π)` 算。这是 `[B, T, A]` 张量。

**为什么非要 log_prob**：PPO 核心是 `ratio = exp(logp_new − logp_old)` 新旧策略概率比。**没 logprob 就没 ratio**，rollout 时必须**当场把采样那一刻的 logprob 存下来**。

**Q：FSDP 跟 DDP 本质区别？为什么 RL 训 VLA 单卡也用？**

详见 [17](17-FSDP框架详解.md)，浓缩版：

**FSDP**：模型 / 梯度 / opt state 按 rank 切片存到不同 GPU，用时 all-gather 凑齐用完释放。等价 DeepSpeed ZeRO-3 的 PyTorch 官方实现。

**跟 DDP 区别**：
- DDP：每卡完整一份 weight/grad/opt，靠 all-reduce 同步 grad。能跑 = 单卡装得下
- FSDP：每卡 1/N 片，forward 前 all-gather，backward 后 reduce-scatter，opt.step 只更新自己片。能跑 = N 卡总和能装下

**显存账**（3B + 4 卡）：
- DDP 每卡 = 6 (bf16 w) + 6 (bf16 g) + 24 (fp32 AdamW) = **36 GB**
- FSDP full_shard 每卡 = 36/4 = **9 GB**

差 4×。3B+ 必 FSDP。

**RL 单卡也用的 4 个理由**（DDP 给不了）：
1. **mixed_precision 统一 API**：bf16 + fp32 reduce + fp32 buffer 一行
2. **offload 钩子**：actor / rollout collocated 互相 offload，FSDP 直接 `flat_param.data.to("cpu")` 整片搬。**RL collocated 核心机制**
3. **gradient checkpointing 边界对齐**：FSDP 知道 wrap unit
4. **可扩展性**：今天 `no_shard`，明天 π0.5 改 `full_shard`，代码不动

---

**Q：FSDP 四种 sharding_strategy 怎么选？**

| 策略 | 切什么 | 显存 | 通信 | 何时 |
|---|---|---|---|---|
| `no_shard` | 啥都不切（≈DDP） | 最高 | 最少 | 单卡，或多卡富余 |
| `shard_grad_op` | grad+opt（≈ZeRO-2） | 中 | 中 | 中等模型，带宽紧 |
| `hybrid_shard` | 节点内 full，节点间复制 | 中 | 跨节点无 | 多节点 + 跨节点带宽差 |
| `full_shard` | param+grad+opt 全切（≈ZeRO-3） | 最低 | 最多 | 大模型，单卡装不下 |

- SmolVLA 单卡：`no_shard`
- π0.5 单节点 4 卡：`full_shard`，前提 NVLink，PCIe 会堵 90%+
- GR00T 多节点：`hybrid_shard` 省跨节点带宽

升 `full_shard` `use_orig_params=True` 别忘改，否则 LoRA init assert。

**Q：VLA + RL 跑起来 success_once 一直是 0 怎么定位？**

**先看视频，再看 tb**。详见 [11 First-line](11-踩坑总结.md)。

tb 上 reward/KL/grad_norm 是**结果**信号；VLA + RL 99% bug 在 adapter 层（image key 路数、state norm、action 单位、起手位 alignment）—— 这些 silent failure 从数值看不出来，**从视频里机械臂形态一眼看出**。

典型形态 → 根因（详见 [11](11-踩坑总结.md)）：

- **几乎不动**：action 单位被乘小，把 normalized 直接 `* π/180`
- **卡奇怪固定姿势**：action 数值过大被关节限位 clip，度数（±100°）当弧度送 SAPIEN
- **完全 random 乱抖**：VLA 输入 OOD
- **一开始就不在 SFT 典型起手位**：env 没 reset 到 SFT keyframe
- **合理但精度差**：sim2real residual gap
- **前 11s 慢慢动 + 第 11s 突然 curl**：SFT 末尾"收 curled rest"，起手位不对前 11s 全在复位

**标准流程**：

```
success_once = 0
  → 1. 打开视频看形态
  → 2. 对照诊断表定位
  → 3. 一次修一项，10-20 epoch 看视频变化
  → 4. 形态对了再调 PPO 超参
```

**反模式**：先调超参 / 只看 tb / 关 video / 多 bug 一起修。

本质：**VLA + RL 核心难点不在 RL 算法，在 model↔env adapter 层；adapter bug 都是 silent failure**。

**Q：debug 一个 RL 不收敛流程？**

按优先级：

1. **先看 reward shape 设计**——很多时候不是 RL 算法，是 reward 错了（reach 太大 → 模型反复 reach 不抓）
2. **rollout 跟 eval policy 输出一致**——同 seed 时确实可复现。我们 SmolVLA 第一次 rollout 和 actor dtype 不一致导致 eval 拿到完全不同 action
3. **advantage 是否合理**——开 `normalize_advantages` 时 std 应在 1 量级，mean ~0；爆掉说明 GAE 算错或 reward 量级离谱
4. **KL 和 entropy**——比 reward 早 expose 问题
5. **最后看 lr / clip_ratio**

**Q：项目最大难点？**

**不是 RL 算法本身**，是 LeRobot 训的 SmolVLA 搬到 RLinf，一开始 SFT eval 成功率几乎 0，逐项对齐 input/output pipeline 把成功率拉起来。

**为什么会**：SFT 模型在它**自己见过的分布**上 work，换环境（LeRobot dataloader → RLinf rollout pipeline），任何一个细节没对齐——image 数量、state 单位、归一化、action 解码——瞬间 OOD。RL 这边没法救：reward 全 0 → advantage 全 0 → 梯度全 0。**RL 能不能涨的前提是 SFT eval 自己得能跑通**，水位达不到调超参没意义。

**具体踩过的对齐点**（按发现顺序）：

1. **image 路数**：SmolVLA config 3 路相机，SFT 靠 `empty_cameras=1` 补占位；RL 漏设 `empty_cameras` → VLM cross-attn token 序列少一路
2. **state normalize**：LeRobot 把 normalize 拆独立 safetensors，`from_pretrained` 不自动加载 → 直接喂 raw qpos 就 OOD
3. **度 vs 弧度**：跟 #2 是两个独立 bug。LeRobot 真机 dataset 是**度**（feetech 协议 ±100°），RLinf/SAPIEN 吃**弧度**。model↔env 边界三处转换：`_initialize_episode` 显式 reset 到 dataset first-frame median 起手位（转弧度）、state 入口 `*180/π`、action 出口 `*π/180`。**少一处机械臂就被关节限位卡住摆奇怪姿势**
4. **action 解码**：RLinf 原版 `prepare_actions_for_maniskill` 写死 EE 7 维（world_vector 3 + rot_axangle 3 + gripper 1），SO-101 是 6 维 joint-pos → 错当 EE 增量发给 sim
5. **reward info key**：`_calc_step_reward` 写死 SimplerEnv info（`is_grasped` / `is_obj_placed`），ManiSkill 这边名字不一样 → RL 拿到 reward 一直 0
6. **chunk_size / action_dim**：SFT 50×6，RL yaml 配错 forward shape 跟 env 期望对不上
7. **dtype**：actor bf16、rollout fp32，同 obs 喂进去 action 差千分之几但 chunk 50 步累积下来轨迹完全不一样

**怎么排查 / 把成功率拉起来**：

- **第一步永远是离线 eval SFT**——独立脚本 load 模型喂手造 batch print 中间值；SFT 仿真里直接跑一轮 eval，至少 20-30% 才往后做 RL。水位达不到说明 adapter 没做好，**先别碰 PPO 超参**
- **每对齐一项重新 eval**：我们 0% → 5%（修 image）→ 25%（修 state norm）→ 40%（修 action 解码）→ 55%（修 reward info key），每修一个都涨一截。修了不涨 → 这项不是瓶颈，回滚
- **凡是"看起来 load 了但其实没"的写脚本验证**：key 对齐、零参数检查、normalize stats 是不是 None，5 行小脚本搞定

**认知层面**：SFT 模型和 RL pipeline 隔着**几十个 implicit contract**（image 数量、归一化、单位、坐标系、chunk、dtype……），LeRobot 自己 dataloader 默认满足，到 RLinf 必须**逐个显式复现**。每个都是 silent failure，**adapter 层工程量比 RL 算法本身大得多**。

**Q：3090 单卡瓶颈？**

显存。SmolVLA 450M base + KV cache 8-10G，actor update 还要 activation buffer，micro_batch 只能 1。bs=1 + 8 envs + 50 chunk 大约 16G。再加 ManiSkill GPU sim 4G。剩 < 2G margin。

速度上 rollout 比 update 慢。`generation_backend: huggingface` 不做 batching 单 env 串行，8 envs × 200 步一轮 5-8 分钟，update 30 秒。**瓶颈在 rollout 推理**。

解：多卡或 SGLang。

**Q：扩展到 π0.5 这种 3B 大模型？**

详见 [14](14-多GPU训大模型.md)：

1. 至少 2 张 24G+ 卡，或 1 张 H100/A100 80G

**Q：sim2real gap 怎么处理？**

仿真先训通，然后：
1. 仿真里加 **domain randomization**：光照、摩擦、相机位置抖、物体颜色随机
2. 真机收 10-50 条轨迹做一轮 BC fine-tune
3. 上真机 RL 时**冻 LoRA**只训 value head 适应新 reward（safer）；或继续 LoRA RL 但 lr 砍到 1e-7
4. 真机 reward：人在 loop 打分（HIL）或预训练 success classifier

我们这次只跑仿真训练，真机部署没做。

**Q：RL 后训练 VLA 最大开放问题？**

按我浅薄理解：

1. **reward 设计不可扩展**：每个任务 hand-craft 没法规模化。LLM RL 用 reward model 学，VLA 还没
2. **sim2real**：仿真训得再好真机 zero-shot 还常崩。world model（SoraVLA / UniSim）还没成熟
3. **多任务 RL**：单任务训出的 policy 没泛化，换物体颜色都失败。multi-task RL 加任务条件但 reward 设计成本爆炸

**Q：YAML 里 reward_type / logprob_type / entropy_type 为什么不一样？**

- `reward_type: chunk_level`：整 chunk（50 step × 6 dim）在 sim 里执行完返回一个 reward
- `logprob_type: chunk_level`：flow_sde 是一次联合 Gaussian 采样，`[T, A]` 是同一"决策"，PPO ratio 必须 sum 成 `[B]`
- `entropy_type: token_level`：entropy 是正则项不进 ratio，保持 `[B, T, A]` 平均更细

不是 bug 是各取所长。OpenPI π0.5 / GR00T 主流 yaml 也是这种组合。

**Q：项目里最隐蔽、最难定位的 bug？**

训了 800 step `success_once` 永远 0，机械臂在仿真乱抖。第一反应 PPO 调参，tb 全异常：grad_norm 1447、EV nan、ratio 2.5。按 PPO 不稳定思路调 lr / clip / entropy 都不行。

停下来想：这些都是**结果**不是**原因**。如果 input 是噪声，PPO 怎么调都错。写独立脚本不走 ray，单 load 模型 + 喂手造 batch print 中间值。两件事一下出来：

1. `policy.config.image_features = {camera1, camera2, camera3}` 三路，但 `empty_cameras=0`。SFT 时 dataset 只 2 路真，靠 `empty_cameras=1` 补一张 -1 + mask=0。RL 时只塞 2 路漏设 `empty_cameras` → VLM cross-attn 视觉 token 序列少一路
2. `policy.normalize_inputs` 是 None。LeRobot 把 normalize 拆独立 `policy_preprocessor.safetensors`，`from_pretrained` 不自动加载。直接喂 maniskill raw qpos（弧度 [-π, π]），但 SFT 是 normalized 度数（57.3× ÷ std）

两个都是 silent failure，没异常 loss 也照下降。**教训：PPO 指标异常时先排查 input pipeline 是否接通再看算法。每个第三方模型有自己 pre/post processor，不能假设 `from_pretrained` 端到端**。

**Q：怎么验证第三方 base 模型权重真加载？**

写 5 行小脚本，**不要信"没报错就 OK"**：

```python
p  = Policy.from_pretrained(base_path)
sd = load_file(f"{base_path}/model.safetensors")

# 1. key 对齐
print('miss in model:', [k for k in sd if k not in p.state_dict()][:5])
print('miss in ckpt :', [k for k in p.state_dict() if k not in sd][:5])
# 2. 全 0 参数
zero = [n for n, t in p.named_parameters() if t.abs().sum() == 0]
print('zero:', len(zero))
```

健康：两个 miss 都空，zero 是 0。

不对齐的情况：
- **JAX/Flax（OpenPI 系）**：key 命名空间完全不同（`/` 不 `.`），kernel `[in,out] → [out,in]` transpose
- **Fork 改 class 名**：state_dict 前缀全变
- **保存时 strip 了 `module.` prefix**

SmolVLA 实测 500 vs 500，0 miss，0 zero。π0.5 那种 JAX 系必查。

**Q：Ray 框架里 debug？pdb 在 worker 里不能用。**

4 个方法按场景选：

1. **离线复现（首选）**：能在主进程复现就别上 ray。独立 python 脚本不导入 ray，构造 fake batch print。10 分钟搞定，比在 ray 挣扎一天强
2. **`remote_pdb`**：worker 里 `import remote_pdb; remote_pdb.set_trace(host='127.0.0.1', port=4444)`，`telnet 127.0.0.1 4444` 接进去
3. **`debugpy` + IDE**：`debugpy.listen(("127.0.0.1", 5678)); debugpy.wait_for_client()`，IDE attach 5678
4. **print / logger**：最低端最稳

第 1 个最有效——很多 bug 看起来在 ray worker，其实根本不依赖 ray。

**Q：state 和 action 都是 6 维，是同一个 6 吗？**

不是。数字撞车，**语义完全不同**：
- **state 6** = 当前 6 关节实测位置，输入（"我在哪"）
- **action 6** = 下一步 6 关节目标位置，输出（"我要去哪"）

能对齐是因为选了 `pd_joint_pos`——action 直接是关节目标角，跟 state 活在同 6 维关节空间。换 `pd_ee_delta_pose` 末端控制，state 还是 6 维关节角，action 变 6 维末端位姿增量，**两个 6 不是一个 6**。

**action chunk 不是 6 维**，是 `[50, 6]`。flow matching 把 300 个数当**一次**联合 Gaussian 采样，PPO ratio 必须 `chunk_level`。

**Q：为什么 SFT 用 LeRobot、RL 用 RLinf，不直接全 RLinf？**

听起来合理但本末倒置：

1. **RLinf 没有 SmolVLA 的 SFT 适配**：扫 `examples/sft/config/` 只有 OpenPI、DreamZero、LingBotVLA。自写 SFT 适配工作量是这次"接 RL"的 2-3 倍
2. **RLinf VLA SFT 当前都 full FT 没 LoRA SFT**：grep 所有 SFT yaml 没有 `is_lora` / `lora_path`。第一个吃螃蟹 bug 不少
3. **LeRobot 才是 SmolVLA 的官方 SFT 框架**：HF LeRobot 团队的模型，整套训练流程（normalize processor、dataset、HIL、calibration、rerun）都是为 LeRobot 量身做的。`_get_default_peft_targets` 直接硬编码在 SmolVLA 源码里。绕开 LeRobot 自造 = 抛弃官方维护链路重造轮子

**正确做法**：SFT 用 LeRobot（官方 + 工具链丰富），RL 用 RLinf（PPO + 分布式 ready），中间一层薄 adapter。adapter 一次性工作量，下次接别的 LeRobot VLA 大部分可复用。

**Q：能不能让 LeRobot 也用弧度跟 RLinf 统一？**

不能，这不是上层框架选的，是底层基础设施差异：

- **LeRobot 必须度**：直接对接 feetech sts3215 真机电机，**度数是电机寄存器协议单位**。改了等于抛弃 HF Hub 上现成的所有 SO-100/SO-101 dataset
- **RLinf 必须弧度**：跑 SAPIEN/MuJoCo，SI 单位。MJCF 第一行 `<compiler angle="radian">`，PD 系数按弧度调

**唯一干净解就是 adapter 层在 model↔env 边界做转换**。三处必改（少一处就废）：`_initialize_episode` 起手位、state 入口 `*180/π`、action 出口 `*π/180`。

少 action 出口的具体后果：VLA 输出 shoulder_lift=-99°，RLinf 当弧度送 SAPIEN（-99 rad = -5673° = 15.7 圈），clip 到 MJCF 限位 ±1.745 rad。**每关节顶到限位**——看起来"模型不会动"，实际被卡住的极端位置。

**接新 LeRobot VLA 5 行验单位脚本**：

```python
info = json.load(open(f"{dataset}/meta/stats.json"))
for key in ["observation.state", "action"]:
    s = info[key]; print(f"{key}: {s['min']} → {s['max']}")
# 绝对值 > 10 → 度；< 4 且接近 ±π → 弧度；≤ 1 → 已 normalize
```

**通用教训**：LeRobot 系 dataset 默认全度数；同类问题在 OpenPI（LIBERO bagging 单位米）、GR00T（每 embodiment 单位不一样）以不同形式出现。**每个 VLA 接 RL 第一件事必查 dataset stats 单位**。

**Q：刚跑起来 success_once 一直是 0，怎么涨上来？**

**第一步永远是 eval SFT，不开 RL**。SFT 模型直接仿真跑 eval，若 success rate=0 那不是 RL 问题是 input/output 没接通——image key 错 / state norm 没加载 / 度弧度没转 / 起手位 / action 解码，任何 silent failure 都让模型 OOD 瞎跑。**SFT eval 至少 20-30% 再开 RL**。

**判断 SFT eval 哪个对齐没做的快速 trick**：看 eval 视频里失败形态——
- 完全伸开/几乎不动 → action 单位 mismatch，关节顶住限位
- 乱抖 random → image key 路数或 state norm 没接通
- 缓慢做合理动作但精度不够 → pipeline 基本对，是 reward shaping 或起手位差几度的微调
- 姿势对但完全在物体旁边乱走 → 起手位 + 底座位置跟真机录数据不一致

SFT eval 过关了再看 RL 这层：

- **reward 多阶段稠密**：reach → grasp → move → on_plate → success，每段都给信号。纯 sparse reward 早期梯度几乎全 0
- **LoRA lr 砍 1e-6**（SFT 1e-4），`clip_ratio_high/low` 收紧 0.15-0.2，`critic_warmup_steps=50`
- **保持探索**：`temperature_train=1.0`、`entropy_bonus=0.005`
- **`total_num_envs` 尽量多**：3090 上限 8，A100 32+

跑起来后**先看 ratio / KL / advantage 早期数值**，比 success rate 早 expose 问题：ratio 飘 1.5+/0.5- → lr 或 clip 不对；KL 飙 → 砍 lr；advantage std 远离 1 → reward 量级离谱；entropy collapse → 探索没了。

**最常卡 success_once=0 的还是 SFT 接进来 silent failure**。RL 早期不涨第一反应应该是"是不是 input 没接通"，不是"PPO 调参有问题"。

**Q：怎么用 ManiSkill 从零搭仿真环境？**

ManiSkill = SAPIEN 物理引擎 + Gym 接口 + GPU 并行 sim 的封装。搭新仿真三层：**Agent（机器人本体）→ Task（场景 + 任务 + reward）→ Env 注册**。

以 SO-101 抓红方块放白盘子为例（详见 [05](05-自定义PickPlace任务.md)、[08](08-代码改动清单.md)）：

**第一层：Agent**（[so101.py](rlinf/envs/maniskill/agents/so101.py)）。继承 `BaseAgent`，交付五样：

1. **模型文件**：MJCF（推荐，比 URDF 干净，物理参数直接 XML 写）或 URDF
2. **keyframes**：标准姿势。`home`、`rest`、`lerobot_start`（对齐 LeRobot 真机 dataset 前 50 ep first-frame median：`[-3.38°, -98.81°, +93.49°, +64.00°, +0.70°, +3.57°]`）。最后这个关键——RL 里 `_initialize_episode` 必须 reset 到它否则起手位跟 SFT 不重合
3. **controller config**：action space。配 `pd_joint_pos`（绝对）/ `pd_joint_delta_pos`（增量）/ `pd_joint_target_delta_pos`。每个 controller 自带 PD `stiffness=1e3, damping=1e2`、力矩限位 100 Nm
4. **sensor config**：相机挂哪。SO-101 挂 `wrist_camera`（`mount=gripper_link` 跟夹爪一起动）；`base_camera`（顶视）由 task 提供
5. **辅助方法**：`is_grasping(obj)`、`tcp_pose`、`is_static`

最后 `@register_agent()` 装饰器挂上，`agent_uid = "so101"`。

**第二层：Task**（[so101_pick_place.py](rlinf/envs/maniskill/tasks/so101_pick_place.py)）。继承 `BaseEnv`，5 个方法：

1. **`_load_scene`**：堆物体（TableSceneBuilder + cube + 自己拼的盘子）
2. **`_initialize_episode`**：每次 reset 随机化。关键三件：机械臂 reset `lerobot_start` keyframe + 微小噪声、plate 固定左下角对齐真机、cube 随机但跟 plate 距离 > 10cm 避免初始 success
3. **`evaluate`**：判 success（xy < 0.057 & z 在凹陷底面 & not is_grasping）
4. **`_compute_dense_reward`**：5 阶段
5. **`_get_obs_extra`**：tcp_pose / obj_pose / plate_pose

**第三层：注册 + YAML**

- `@register_env("SO101PickPlace-v0", max_episode_steps=400)`
- YAML 里 `env_id / robot_uids / control_mode / num_envs / sim_backend: gpu`
- `gym.make(env_id, num_envs=8, ...)` 直接 step

**ManiSkill 免费红利**：GPU 并行 sim（`num_envs=8` 同步 step 占 4G）、obs 自动打包 dict、action space 自适应 controller。

**一句话**：Agent 定义"机器人是什么"，Task 定义"场景 + 任务 + reward + success"，注册一下用 Gym 接口 step。复杂度全在 Task 的 reward 设计 + 初始状态分布跟真机对齐。

**Q：SmolVLA SFT 三路相机但仿真两路怎么处理？**

详见 [11 #0.A](11-踩坑总结.md)。

`SmolVLAConfig.image_features` 硬编码 3 路（camera1/2/3），SFT dataset 只 2 路真（front+wrist），靠 `empty_cameras=1` 自动补一张全 -1 + mask=0 占位。

仿真也 2 路，直接照搬：
- maniskill `base_camera` → `observation.images.camera1`
- maniskill `wrist_camera` → `observation.images.camera2`
- 不传 `camera3`，`empty_cameras=1` 自动补 0

**为什么补 0 不掉点**：mask=0，cross-attn 看不到。SFT 训练时就这分布，RL 完全 in-distribution。

凑齐第三路真图（如 overhead）反而引入新视角 OOD，比补 0 还差。除非重 SFT。

**Q：从机械臂 VLA+RL 过渡到轮式或双足？**

先声明：自己没真训过轮式和双足，下面是基于机械臂推过去的判断。

**能直接搬（VLA+RL 方法论本身和形态无关）**：
- 训练 pipeline：SFT 冷启动 → RL 后训练
- RL 算法：PPO + GAE + value head、KL 监控、entropy
- VLA 架构范式
- RLinf 4 角色 + FSDP + weight sync
- sim2real 思路：DR、真机少量 fine-tune、HIL reward

**必须重新设计**：

1. **action space**：
   - 机械臂：6-7 DoF (joint pos / EE delta)，chunk 50 step
   - 轮式：2 DoF 连续（v, ω）或差速 (v_l, v_r)，chunk 短
   - 双足：**维度爆炸**腿 ~12 + 躯干 + 手臂 ~14 DoF。**关键认知**：VLA **不直接出关节扭矩**，而是出 high-level command（root velocity / target footstep / base orientation）给 **WBC (whole-body controller)** 翻译成扭矩。**VLA 30-50Hz，WBC 500-1000Hz，分层架构**——跟机械臂端到端两种范式

2. **observation 重心转移**：
   - 机械臂：视觉主，proprio 辅
   - 轮式：第一人称视觉 + odometry/IMU，可能 depth/LiDAR
   - 双足：**proprio 反客为主**——IMU、关节位置速度、足底接触力直接决定能否站住

3. **控制频率层级化**：机械臂 30Hz 单层；双足必分层 VLA 30Hz 出 high-level，WBC 1kHz 出 torque。**两层接口设计追问点**——target velocity 还是 target footstep？

4. **reward 换思路**：
   - 机械臂：5 阶段稠密线性
   - 轮式：`-distance + collision + smoothness`，可能 social-aware
   - 双足：**不摔倒 hard constraint**。`任务 + alive bonus − 扭矩平方 − 步态对称 − foot slip − air time`，至少 6-8 项加权。比机械臂复杂一个量级

5. **仿真器换**：ManiSkill → Habitat/iGibson/Isaac Sim → **Isaac Lab/MuJoCo MPC/Genesis**。双足对接触动力学要求最高

6. **sim2real gap 量级**：机械臂小 → 轮式中 → 双足**最大**，必须 **massive DR**（成千上百参数组合并行训）+ **teacher-student 蒸馏**

**一句话给面试官**：方法论、训练框架、算法实现层基本能复用，"接什么 model + 配什么 reward + 用哪个仿真器 + 是否要分层 controller" 才是真要重做的。**最关键认知差异：双足必须 VLA + WBC 分层，不能端到端直接出关节角度**。

**我得补的短板**：WBC/MPC 基础（OCS2/Pinocchio）、双足步态 reward shaping、Isaac Lab 大规模并行（4096 envs）。轮式 VLN 跟我现在 VLA 思路最近过渡成本最低。
