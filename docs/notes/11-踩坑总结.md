# 11 · 踩坑总结

按"重要度 × 复现概率"排序。每条：症状 → 原因 → 修复。

## 🔍 First-line：先看视频，再看 tensorboard

**接 VLA 到 RL 框架最高 ROI 的调试技巧**。yaml 里 `save_video: True` 一定开。Tensorboard 是结果信号（"事情不对"），视频是即时信号（"为什么不对"）。**一段 13 秒视频比 10 小时调 PPO 超参更快定位根因**。

### 视频形态 → 根因诊断对照

| 视频形态 | 根因 | 排查 |
|---|---|---|
| **几乎不动 / 动作幅度极小** | action 单位转换被乘小 —— 把 normalized (~N(0,1)) 直接 `* π/180`，每关节 target 只 0.001-0.05 rad，PD 几乎不发力 | 0.B normalize + 0.D 度/弧度 |
| **卡在奇怪的固定姿势**（顶到限位） | action 过大被关节限位 clip —— 度数（±100°）当弧度送 SAPIEN = 5700° = 16 圈 | 0.D 度 vs 弧度 |
| **完全 random 乱抖** | VLA 输入 OOD —— 图像 key 路数 / state 没 normalize / dtype | 0.A 图像、0.B normalize、3 dtype |
| **第 0 ep 起手对，第 1 ep+ 起手飘** | `auto_reset` 路径没正确调 `_initialize_episode` | 0.D auto_reset 静默退化 |
| **一开始就不在 SFT 典型起手位** | env 没 reset 到 SFT keyframe，留在 MJCF 默认 qpos | 0.D 起手位 keyframe |
| **合理但精度差** | sim/real residual gap | sim2real、reward |
| **越走越飘** | 数值不稳定 —— dtype 累积 / value 没收敛 / KL 太快 | 3 dtype、PPO 超参 |
| **前 11s 慢慢动 + 第 11s 突然 curl** | SFT 末尾 "放方块 → 收 curled rest"；起手不对前 11s 全在复位 | 修起手位后这是任务后的自然结束 |

### 标准排查流程

```
training 50+ epoch success_once 仍 0
  ↓ 1. 打开训练视频，看机械臂形态
  ↓ 2. 对照表初步定位
  ↓ 3. Tier 0 / Tier 1 具体条目
  ↓ 4. 修一项跑 10-20 epoch，看视频形态变化 + reward
  ↓ 5. 视频形态对了再调 PPO 超参
```

**反模式**：
- ❌ 先调 PPO 超参（adapter 没修好再好的超参也救不回来）
- ❌ 只看 tb 不看视频
- ❌ 关 video 省磁盘
- ❌ 多 bug 一起修（不知道哪个起作用）

## ⭐⭐⭐ Tier 0：训练静默失败（最危险）

模型不报错，loss 在动，但学的是噪声。`success_once` 永远 0。根因都在 LeRobot pre/post processor 没接通。

### 0.A `image_features` 跟 batch key 不匹配 → VLM 瞎

**症状**：能跑 loss 动，`success_once` 800 step 仍 0，视频里 random 抖。tb 看到 grad_norm=1447、critic EV=nan、ratio=2.5。

**原因**：`SmolVLAConfig.image_features` 是 `{camera1, camera2, camera3}` 三路，SFT 时 dataset 只有 2 路真（front + wrist），靠 `empty_cameras=1` 补一张 -1 + mask=0 占位。RL 接入只塞 2 路漏占位，`empty_cameras=0`（base 默认），VLM cross-attn token 序列比训练时少一路。

**修复**：

```python
# rlinf/models/embodiment/smolvla/__init__.py get_model 内，PEFT load 之后
base_policy.config.empty_cameras = 1
```

**为什么补 0 不影响精度**：LeRobot 给 empty camera 塞 `attention_mask=0`，cross-attn 完全看不到这一路。SFT 训练时就这样，RL 复用完全 in-distribution。

**通用教训**：接 LeRobot 模型时先 `print policy.config.image_features` 看要几路，对照 dataset 实际几路真相机，差值就是 `empty_cameras`。`from_pretrained` 默认重置 `empty_cameras=0`。

### 0.B normalize_inputs / unnormalize_outputs 没加载 → state 量级 + 单位双错

**症状**：同 0.A，看不出来；和 0.A 一起更难定位。

**原因**：LeRobot 把 normalize 拆到独立的 `policy_preprocessor_step_5_normalizer_processor.safetensors`，要走 `make_smolvla_pre_post_processors` pipeline 才挂上。`SmolVLAPolicy.from_pretrained` **只加载模型权重，不加载 normalize stats**。

dataset stats（degrees）：

```python
sd['action.mean'] = [-10.3, 13.2, -8.9, 72.9, 2.5, 12.7]
sd['action.std']  = [42.8, 41.2, 46.5, 16.7, 20.5, 12.3]
```

SFT 训练时 normalize 到 N(0,1)。我们仿真给 `env.unwrapped.agent.robot.get_qpos()` 是弧度 `[-π, π]`。**三错**：单位错（弧度 vs 度，差 57.3×）、量级错（N(0,1) vs raw）、输出错（normalized action 没 unnormalize 直接送 controller）。

**修复 —— 4 步必做**：

1. **`get_model` load 两个 safetensors → buffer**：

   ```python
   pre  = load_file(f"{lora_path}/policy_preprocessor_step_5_normalizer_processor.safetensors")
   post = load_file(f"{lora_path}/policy_postprocessor_step_0_unnormalizer_processor.safetensors")
   model.register_buffer("state_mean",  pre["observation.state.mean"].float(),  persistent=False)
   model.register_buffer("state_std",   pre["observation.state.std"].float().clamp(min=1e-8),  persistent=False)
   model.register_buffer("action_mean", post["action.mean"].float(),  persistent=False)
   model.register_buffer("action_std",  post["action.std"].float().clamp(min=1e-8),  persistent=False)
   ```

2. **state 入口（`_build_lerobot_batch`）`rad → deg → normalize`**：

   ```python
   state = env_obs["states"][..., :6].float() * (180.0 / math.pi)
   state = (state - self.state_mean) / self.state_std
   batch[OBS_STATE] = state.to(dtype=param_dtype)
   ```

3. **action 出口（`predict_action_batch`）`unnormalize → deg → rad`**，但 `forward_inputs["action"]` 保留 normalized（PPO 在 normalized 域算 Gaussian）：

   ```python
   actions_norm = outputs["actions"][:, :chunk, :env_dim]
   actions_deg  = actions_norm * self.action_std + self.action_mean
   actions      = actions_deg * (math.pi / 180.0)     # env 用 rad
   forward_inputs = {..., "action": actions_norm.reshape(B, -1).contiguous(), ...}
   ```

4. **`action_utils.py` so101 分支直通**，不在这里转换：

   ```python
   if "so101" in policy:
       return raw_chunk_actions
   ```

**坑中坑**：早期看到 dataset action 范围 ±100° 就在 `action_utils.py` 加 `* π/180`。**错的**。LeRobot 模型输出是 **normalized 域**，要先 `* std + mean` 拿回度数才能 `* π/180`。

**通用教训**：
- LeRobot 系（SmolVLA / ACT / pi0_lerobot）全用独立 `policy_preprocessor.safetensors` + `policy_postprocessor.safetensors`，`Policy.from_pretrained` 不会自动加载
- `register_buffer` 不够，还要在 `forward / sample_actions` 前后**实际应用**
- 看 dataset action 范围 ≠ 看 model 输出范围。dataset 是 raw 度数，model 是 normalized
- 检查 `policy_preprocessor.json` 的 `norm_map`，确认 STATE 和 ACTION 用 `MEAN_STD / MIN_MAX / IDENTITY` 哪种

### 0.C state padding 顺序对，因 0.B 失效

LeRobot 顺序是 normalize（前 6 维有 stats）→ pad 到 `max_state_dim=32`（后 26 维 0）。顺序本身对，但 0.B 让 normalize 没生效，等于 raw qpos + 26 个 0 喂模型。**修完 0.B 自动好**。

### 0.D 度 vs 弧度：跨生态单位 mismatch

**症状**：`success_once` 800 step 仍 0，机械臂在视频里**奇怪伸开几乎不动**；reward 横盘 0.05-0.10。

**和 0.B 关系**：0.B 是"normalizer 没加载"，0.D 是"normalizer 加载了，但喂进 normalizer 的量本身就是错单位"。**必须都修才能 work**。

**根本原因**：跨生态底层差异：

| 生态 | 原生单位 | 为什么 | 能改吗 |
|---|---|---|---|
| LeRobot dataset | 度（含 gripper [0,42°]） | 直接对接 feetech sts3215 真机电机寄存器协议 | 否 |
| SAPIEN / MuJoCo / `pd_joint_pos` | 弧度 | SI 单位，PD 系数按弧度调 | 否 |

**结论**：必须在 model↔env 边界做转换。**三处必改**：

| 边界 | 原方向 | 修法 |
|---|---|---|
| **起手位**（env reset → 仿真初始 qpos） | TableSceneBuilder 没 so101 分支，留 MJCF 默认 → 跟 LeRobot 录数据姿势完全不同 → 第一帧 OOD | `_initialize_episode` 显式 `agent.reset(qpos)`，qpos 用 dataset first-frame **主聚类 mean** 转弧度（不能简单 median：dataset 经常多段录制合并，多峰分布 median 落到过渡区） |
| **state 入口**（env→VLA） | `flatten_state_dict` 出 22 维弧度（qpos[6] + qvel[6] + tcp[7] + plate[3]），SFT 见的是 6 维度数 | 切前 6 维 + `* 180/π` |
| **action 出口**（VLA→env） | SmolVLA 输出度数 chunk，`action_utils.py` 原本直通 | `* π/180` 转弧度（不！0.B 修法是在 wrapper 内 unnormalize+转换，action_utils 保持直通） |

**症状细节**：action 出口没转 → VLA 输出 shoulder_lift=-99° → SAPIEN 当弧度（-99 rad = -5673° = 15.7 圈）→ `pd_joint_pos` clip 到限位 ±1.745 rad → 每个关节顶到限位上 → 视觉看到"摆奇怪固定姿势"。

**dataset 单位验证脚本**（接任何新 LeRobot VLA 前必跑）：

```python
import json
info = json.load(open(f"{dataset}/meta/stats.json"))
for key in ["observation.state", "action"]:
    s = info[key]
    print(f"{key} range: {s['min']} → {s['max']}")
# 绝对值 > 10           → 度（feetech 协议）
# 绝对值 < 4 且接近 ±π   → 弧度
# 绝对值 ≤ 1            → 已 normalize
```

**起手位分布检查**（验单位后立刻跑，避免多峰把 keyframe 设到过渡区）：

```python
import pyarrow.parquet as pq, glob, numpy as np
df = pq.read_table(sorted(glob.glob(f"{dataset}/data/chunk-*/*.parquet"))[0]).to_pandas()
firsts = df[df.frame_index == 0]
states = np.array(firsts["observation.state"].tolist())
print("first-frame std:", np.round(states.std(axis=0), 2))   # >20° → 多峰
# 名字带 combined / merged / mixed 一律先聚类，从 KMeans 取主簇 mean
```

**二阶坑**：选 keyframe 后必须验 `keyframe ± init_qpos_noise` 在 MJCF 限位内。SO-101 curled 姿势 `shoulder_lift=-98.81° ≈ -1.725 rad`，距限位 -1.745 rad 只剩 0.02 rad ≈ 1.15°；ManiSkill `robot_init_qpos_noise` 默认 0.02 rad → 50% 概率超限。我们用 0.003 rad。**接新机器人必算 noise vs limit margin**。

**三阶坑（auto_reset 静默退化）**：`_initialize_episode` 给 `agent.reset(qpos)` 传 batched `qpos[b, 6]` 第一次 manual reset 能 work，但后续 `auto_reset` 走 partial reset 时 ManiSkill GPU sim 跟 reset_mask 协议对不上，keyframe 没真正写进 GPU 缓冲。

症状：`0.mp4` 起手是 curled ✓，但 `1.mp4` 起手回到 extended（实际是上 ep 末态）。每个视频到 ~11s 模型"突然回 curled" —— SFT 末尾的自然行为，但因为前 11s 都在复位最后 2s 没时间。

**修复**：照 [put_carrot_on_plate.py:62-94](../../rlinf/envs/maniskill/tasks/put_carrot_on_plate.py#L62-L94) canonical 姿势：

```python
self.agent.robot.set_pose(sapien.Pose([0.0, 0.0, 0.0]))     # 先 base pose
start_qpos = self.agent.keyframes["lerobot_start"].qpos      # [6] unbatched
if self.robot_init_qpos_noise > 0:
    noise = self._episode_rng.normal(0, self.robot_init_qpos_noise, len(start_qpos))
    start_qpos = start_qpos + noise                          # 保持 1D [6]
self.agent.reset(init_qpos=start_qpos)                       # unbatched → 广播
```

**通用教训**：ManiSkill `_initialize_episode` 给 robot 设 qpos **永远用 unbatched 1D 广播**，不要 batched `[b, dof]`。物体位置用 `[b, 3]` batched OK（走 `actor.set_pose`）。

### 0.E Real-to-Sim 视觉领域鸿沟 (Domain Gap) → 起点即终点的 0% 成功率

**症状**：代码完美无报错，单位和维度检查毫无瑕疵，但从 Step 1 起 return 极低（例如仅拿到微弱的 `reach_r` ≈ 0.09），无论跑多久 `success_once` 永远 0。

**和前序 bug 关系**：当你把 `empty_cameras` (0.A) 和 单位/Normalizer (0.B/0.D) 都修对后，依然遭遇开局 0%，唯一合理的解释。

**根本原因**：你使用的是 **纯真机数据微调出的 SFT 模型**（如 `smolvla_lora_real_only`），但尝试将其应用于**光影纯净的 SAPIEN 仿真环境**中（无反光、无噪点、单色背景）。
1. VLA 大脑陷入严重的 **视觉 OOD（分布外）**，由于没见过 SAPIEN 画风的方块和盘子，开局当场“失明”。
2. PPO 陷入 **探索瓶颈（Exploration Bottleneck）**：强化学习能学好的前提是“偶尔能撞对并拿到大额的抓取高分”。如果因为视觉断层导致模型根本摸不到方块边缘，模型就会非常“聪明且偷懒”地退化：停在起手位附近轻轻晃动，永远只拿安全保底的靠近分（0.09/step）。

**修复 / 验证 —— 发现即终结问题**：
1. **零样本评估排雷（极度重要！）**：
   在启动任何带有 `ppo_` 字眼的干预任务前，**必须先通过纯推理验证 SFT 的零样本迁移能力**。RLinf 提供了完善的评估旁路，停掉所有 RL 梯度与探索噪声：
   ```bash
   bash examples/embodiment/eval_embodiment.sh [你的配置文件如: maniskill_ppo_smolvla_so101]
   ```
   **必做检查**：运行完毕后去 `video/eval/` 目录查看渲染出的 mp4 视频。如果模型在没加 PPO 的时候表现为“呆若木鸡”、“原地轻微抽搐”或“朝着完全相反的虚空方向抓取”，说明遭遇了致命的 Real-to-Sim Gap，PPO 根本没有优化生长的地基。
   
2. **两大解法路线**：
   - **算法线（推荐）**：回撤到数据准备阶段，加入一定比例的 SAPIEN 仿真数据进行 Sim Co-training（真+仿混合训练）。只有 SFT 见过了 SAPIEN 画风，后续的 PPO 才能长出翅膀。
   - **环境线**：在 SAPIEN 里做极高的 Domain Randomization，强行把背景墙和地板贴图全换成真机桌面的高清图片材质，并引入强光照随机性去骗过 VLA 的眼睛。

## ⭐⭐⭐ Tier 1：架构级，第二次接模型还会踩

### 1. PEFT LoRA 必须包顶层 SmolVLAPolicy

**症状**：`PeftModel.from_pretrained` target_modules 没匹配，或 LoRA 加载完 trainable 显示 0。

**原因**：SFT `adapter_config.json` 的 target_modules 正则是 `model\.model\.vlm_with_expert\..*?\.q_proj`，带 `model.` 前缀。包到 `base_policy.model` 少一层 `model.`，全部不匹配。

**修复**：

```python
base_policy = SmolVLAPolicy.from_pretrained(cfg.model_path)
base_policy = PeftModel.from_pretrained(base_policy, lora_path, is_trainable=True)
# ↑ 包顶层，不是 base_policy.model
```

**通用教训**：换模型先看 adapter_config target 正则前缀，决定 wrap 在哪一层。

### 2. FSDP + LoRA 必须 `use_orig_params=True`

**症状**：`ValueError: Must flatten tensors with uniform requires_grad when use_orig_params=False`

**原因**：FSDP 默认把同 unit 所有 param flatten 成连续内存，要求 `requires_grad` 一致。LoRA 训练里 base=False，LoRA=True，不能 flatten。

**修复**：

```yaml
fsdp_config:
  use_orig_params: True
```

**副作用**：开后 `.parameters()` 返回原始 fp32 张量，不是 FSDP cast 后的 bf16。`next(model.parameters()).dtype` 是 fp32，按这个 cast 会跟实际 forward 时 bf16 输入对不上 → 下面 dtype 坑。

### 2.5 FSDP1 + `use_orig_params` + 冻结/LoRA 混合 → `_writeback_orig_params` shape mismatch

**症状**：解决 #2 之后，第一次 actor forward：

```
RuntimeError: Cannot writeback when the parameter shape changes
Expects torch.Size([589824]) but got torch.Size([768, 3, 16, 16])
```

`589824 = 768×3×16×16` 是 SigLIP `patch_embedding` Conv2d weight 的 numel。

**根本原因**：PyTorch FSDP1 `use_orig_params=True` 在 `_use_unsharded_flat_param` 里 forward 阶段用 `as_params=False`，把 `module.weight` 从 `nn.Parameter` 替换成普通 `Tensor`（autograd 需 Tensor 视图能进计算图）。

- **trainable 参数**：backward 完后会走 `as_params=True` 恢复 Parameter，下次 `param_changed=False`，安全
- **冻结参数**（vision tower 等）：不参与 backward → view 不被恢复 → 下次 `param_changed=True` → 进 1D vs 4D 比较 → **必报错**

跟 PyTorch 版本无关（2.0-2.7 都这样），跟 peft / transformers 版本也无关。

**二阶问题**：只排除冻结 base 不够。LoRA trainable 叶子也会和 root 一起塞进同一 FlatParameter，在 corner case 下也会撞同样 raise。

**修复 —— 双管齐下**：

**第 1 步**：[smolvla/__init__.py](../../rlinf/models/embodiment/smolvla/__init__.py) `get_model` 末尾 DFS 收集"内部完全冻结"的最顶层子树，挂到 model：

```python
if train_lora_only:
    ignored = []
    def _visit(mod):
        params = list(mod.parameters(recurse=True))
        if not params:
            return
        if all(not p.requires_grad for p in params):
            ignored.append(mod); return  # 整子树冻结，不再下钻
        for child in mod.children():
            _visit(child)
    for child in model.children():
        _visit(child)
    model._fsdp_ignored_modules = ignored
    model._fsdp_force_lora_wrap = True
```

**第 2 步**：[fsdp.py wrap_model](../../rlinf/hybrid_engines/fsdp/strategy/fsdp.py) 强制启用 LoRA wrap_policy，把每个 trainable 叶子独立 wrap 成 FSDP 单元，并把 `_fsdp_ignored_modules` 传 FSDP：

```python
is_lora_for_wrap = bool(self.cfg.model.is_lora or getattr(model, "_fsdp_force_lora_wrap", False))
auto_wrap_policy = get_fsdp_wrap_policy(module=model, config=self.cfg.fsdp_config,
                                       is_lora=is_lora_for_wrap, model_type=self.cfg.model.model_type)
ignored_modules = getattr(model, "_fsdp_ignored_modules", None) or None
fsdp_model = FSDP(module=model, ..., ignored_modules=ignored_modules)
```

**为什么用独立 `_fsdp_force_lora_wrap` 而不是 `cfg.is_lora`**：SmolVLA 因 #1 必须把 `cfg.is_lora` 设回 False 防外层二次 wrap，结果 wrap_policy 也跟着关。独立标志解耦"外层 PEFT 开关" 和 "wrap policy 开关"。

**最终架构**：
- 冻结 base（~450M）→ `ignored_modules`，FSDP 不管
- LoRA A/B + value_head Linear → 每个独立 FSDP 单元
- root unit 几乎空，writeback 不再扫到 4D 冻结参数

**为什么其它 yaml 没撞**：路径选择不同：

| 模型 | 训练 | `cfg.is_lora` | `use_orig_params` | wrap_policy LoRA | 撞 #2.5 |
|---|---|---|---|---|---|
| OpenPI / GR00T / DexBotic 等 | full FT | False | False | 不需要 | ❌ 没冻结/trainable 混合 |
| OpenVLA / OFT（13 份 yaml） | LoRA | **True** | False | ✅ 每 trainable 叶子独立 unit | ❌ |
| **SmolVLA** | LoRA | **被迫 False** | True（被迫） | ❌ 跟着关 | **✅** |

**OpenVLA-OFT 关键经验**：fsdp_config 极简，没 `use_orig_params`、没 `ignored_modules`，全靠 wrap_policy LoRA lambda 给每个 trainable 叶子独立 wrap → 单元内 `requires_grad` 统一 → `use_orig_params=False` 直接 flatten，**绕过 #2 和 #2.5 整条 bug 链**。我们的双标志修法本质就是**手动复刻 OpenVLA-OFT 效果**。

**通用教训**：
- FSDP1 + `use_orig_params=True` 对**任何冻结 + trainable 混合**训练都有此潜在 bug，不限于 SmolVLA
- 接"PEFT 自己跑"的模型时，分清 `cfg.is_lora` 是给"外层 get_model" 还是给 "wrap_policy" 用，要独立标志
- 根治是切 FSDP2（per-parameter sharding，无 FlatParameter 簿记），但 RLinf FSDP2 路径要补适配
- 不要先去查 PyTorch 版本 —— 跟版本无关

### 3. dtype mismatch in linear forward

**症状**：`RuntimeError: mat1 and mat2 must have the same dtype, but got Float and BFloat16`，常在 value_head 或 action_out_proj。

**根因**：FSDP cast 输入 bf16；value_head weight 在 `param_dtype: bf16` 下也是 bf16；但中间手动 `.to(fp32)` 转 fp32 → mat1 (fp32) × mat2 (bf16) 报错。

**正确姿势**：两条路二选一不混用：
- **路 A：fp32 weight + amp_autocast**（OpenPI）：`param_dtype: null`，外面包 `torch.amp.autocast(bfloat16)`
- **路 B：bf16 weight + 出口 cast**（我们，3090 显存吃紧）：`param_dtype: bf16`，wrapper 内全 bf16，return 前 `.float()`

**我们的修法**：

```python
model = SmolVLAForRLActionPrediction(lerobot_policy=base.to(bf16), config=cfg).to(bf16)
# forward 出口：
return {"logprobs": lp.float(), "values": v.float(), "entropy": h.float()}
```

**教训**：全链路统一一种 dtype；只在两个边界做 cast：(1) 整体 `.to(bf16)`；(2) PPO loss 入口 `.float()`。

### 4. peft 不能 > 0.18

**症状**：`ImportError: cannot import name 'EmbeddingParallel' from 'transformers.integrations.tensor_parallel'`

**原因**：peft 0.19+ 检测 `torch.distributed.is_initialized()` 走 TP sharding 分支，import transformers 4.57.x 还没合并的符号。

**修复**：`pip install peft==0.17.1`（RLinf dreamzero 配方）。

### 4.5 Base 权重 key 前缀对齐 —— 接新模型必查

**判断要不要 key rename**：

| 来源 | rename | 例 |
|---|---|---|
| LeRobot `save_pretrained` 写的 PyTorch | 不用 | SmolVLA |
| OpenPI 官方 JAX/Flax pickle | 必须 | π0、π0.5 |
| HF Hub 原生 transformers | 一般不用，仓库 fork 改过类名要 | OpenVLA-OFT |

**验证脚本**：

```python
from safetensors.torch import load_file
p  = Policy.from_pretrained(base_path)
sd = load_file(f"{base_path}/model.safetensors")
print('miss in model:', [k for k in sd if k not in p.state_dict()][:5])
print('miss in ckpt :', [k for k in p.state_dict() if k not in sd][:5])
zero = [n for n, t in p.named_parameters() if t.abs().sum() == 0]
print('zero-init:', len(zero), zero[:5])
```

健康：两个 miss 都 0，zero 是 0。SmolVLA 实测 500 vs 500，0 miss，0 zero。

对比 π0.5：JAX key 是 `PaliGemma/img/.../query/kernel`，PyTorch 期望 `paligemma_with_expert.paligemma.vision_tower.....q_proj.weight`。完全不一样，必须写 mapping + kernel `[in, out] → [out, in]` transpose。

### 5. LeRobot SmolVLAConfig 不是 HF PretrainedConfig

**症状**：peft `_get_tied_target_modules` 报 `AttributeError: 'SmolVLAConfig' object has no attribute 'get'`

**原因**：LeRobot 把 config 写成 `@dataclass`，没继承 HF `PretrainedConfig`，没 `to_dict()` 也没 `.get()`。

**修复**：

```python
if not hasattr(cfg_obj, "to_dict"):
    cfg_obj.to_dict = lambda: {"tie_word_embeddings": False}
```

**教训**：第三方 VLA 库（LeRobot / OpenPI / GR00T）经常改 config / model 基类，接入 RLinf 几乎一定要补 adapter。

## ⭐⭐ Tier 2：环境层

### 6. `prepare_actions` 对非 EE 控制需 bypass

**症状**：`KeyError: 'gripper'` in `prepare_actions_for_maniskill`。

**原因**：默认 maniskill EE 控制，要把 `[B, T, 7]` 解析成 `world_vector(3) + rot_axangle(3) + gripper(1)`。SO-101 是 joint-pos 6 维。

**修复**：

```python
if "so101" in policy:
    return raw_chunk_actions
```

### 7. RLinf 默认 reward 用 Google/WidowX info key

**症状**：`KeyError: 'is_src_obj_grasped'` / `'consecutive_grasp'` in `_calc_step_reward`。

**原因**：默认 reward mode 假设 SimplerEnv info（`is_src_obj_grasped`、`consecutive_grasp`）。SO101 task 返回的是 `is_grasped / cube_on_plate / success`。

**修复**：env yaml 设 `reward_mode: "raw"`。

### 8. wrap_obs_mode 决定相机命名

**症状**：rollout 启动后 obs dict 缺 `main` key 或 `extra_view_images` 空。

**原因**：`_wrap_obs` 硬约定：`default` → `3rd_view_camera`；`simple` → `base_camera`，其他进 `extra_view_images`。

**修复**：third-person 相机命名 `base_camera`，wrist 随便（字典序进 extra），yaml `wrap_obs_mode: simple`。

## ⭐ Tier 3：配置写错

### 9. mixed_precision 用 RLinf 命名

**症状**：`ValueError: Could not parse the precision of 'bfloat16' to a valid torch.dtype`

**修复**：用 RLinf 内部名 `bf16 / fp16 / fp32 / null`，不是 torch 的 `bfloat16 / float32`。

### 10. sharding_strategy 大小写敏感

**症状**：`AssertionError: Unknown sharding strategy: NO_SHARD`

**修复**：用 `no_shard`（小写）。

### 11. flow VLA 必须 chunk_level log_prob

**症状**：训得动但 ratio 在 [1e-5, 1e5] 大幅震荡，loss 爆炸。

**原因**：flow_sde 是"一次联合 Gaussian 采样"出 `[B, T, A]=300` 个值。`token_level` 把它当 300 个独立决策 clip，违背"一次采样一个决策"。

**修复**：

```yaml
algorithm:
  logprob_type: chunk_level
  reward_type: chunk_level
  entropy_type: token_level
```

### 12. flow VLA 不需要 max_new_token

**症状**：`ConfigAttributeError: Key 'length_params' is not in struct`

**修复**：

```yaml
length_params:
  max_new_token: null     # flow VLA 不生成 token
  max_length: 1024
  min_length: 1
```

### 12.5 `max_episode_steps` 必须整除 `chunk_size`

**症状**：训练能跑但行为奇怪 —— 最后一个 chunk 输出被截断、buffer 长度不一致、`generate_one_epoch` shape 错误。

**原因**：flow VLA action 按整 chunk 生成 + 执行（`huggingface_worker.py` 一次 forward `[B, chunk, action_dim]`）。env 每次接收完整 chunk 执行 `chunk_size` 步物理。不能整除 → 最后一个 chunk 不完整，行为依赖代码版本（被拒收 / 强制执行完 / silent 退化）。

**修复**：选能整除的值（30 FPS 时长换算）：

| `max_episode_steps` | chunks | 时长 |
|---|---|---|
| 200 / 250 / 300 / 350 | 4 / 5 / 6 / 7 | 6.67-11.67 s |
| **400** | **8** | **13.33 s（这次任务用）** |
| 500 | 10 | 16.67 s |

**3 处同步改**（漏一处自坑）：

1. **task 注册**：`@register_env("SO101PickPlace-v0", max_episode_steps=400)`
2. **env yaml**：`max_steps_per_rollout_epoch: 400` + `max_episode_steps: 400`
3. **主 config**：`env.train` 和 `env.eval` 各一份

**怎么定具体值**：dump 真机 dataset `frame_index` 分位数，选覆盖 **q90+ 留 margin** 的最小整除值（RL 阶段动作不熟练，覆盖 q99 保险）：

```python
sd = load_file('<ckpt>/policy_preprocessor_step_5_normalizer_processor.safetensors')
for k in ['frame_index.q10', 'frame_index.q50', 'frame_index.q90', 'frame_index.q99', 'frame_index.max']:
    print(f'{k:25s} = {sd[k].item()/30:5.2f} 秒')
```

### 13. micro_batch_size > 1 在 3090 OOM

**原因**：SmolVLA forward 显存按 batch 线性，bs=2 bf16 activation ~18GB > 22GB 可用。

**修复**：`micro_batch_size: 1` + `update_epoch: 4` 拉满更新次数。

### 14. value_head `bias_last` 影响数值稳定

**原因**：MLP 最后一层带 bias 能吸收 target mean。OpenPI 用 `bias_last=True`，我们对齐。

## 排查 dtype 问题通用流程

1. **打印整条链路 dtype**：obs → embed → expert → suffix → projection → output 每步 `print(x.dtype, w.dtype)`
2. **看 FSDP wrap 之后**：FSDP 按 `param_dtype` 自动 cast input，断点常在 FSDP unit 边界
3. **`use_orig_params=True` 下 `.parameters().dtype` 撒谎**：要看实际行为去看 `.weight.dtype`
4. **解决思路：全链路统一 + 出口 cast**，不要中间某层"修一下"
